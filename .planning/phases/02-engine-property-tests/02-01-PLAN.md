```rust
    fn id(&self) -> &'static str;
    fn priority(&self) -> P;
    fn before(&self, _state: &S, _event: &mut E, _tx: &mut Transaction<O>) {}
    fn after(&self, _state: &S, _event: &E, _tx: &mut Transaction<O>) {}
}
```

Key behavioral notes (from engine.rs source):
- dispatch only pushes CommitFrame when `tx.irreversible && !tx.cancelled`
- dispatch only updates replay_hash when `tx.irreversible && !tx.cancelled`
- Transaction::new() defaults: irreversible=true, deterministic=true, cancelled=false
- Turns(n) is decremented in the post-dispatch loop — runs unconditionally (even on cancelled tx)
- Triggers(n) is decremented inside the before() loop — only when rule.id() is in enabled
- engine.enabled and engine.lifetimes are PRIVATE — verify lifetime/enabled restoration indirectly
- dispatch_preview saves and restores state, lifetimes, enabled, and replay_hash — all four fields

**PROP-02** (`prop_02_preview_isolation`):
- Strategy: `op_sequence_strategy()` (same as PROP-01)
- Create engine with `NoRule` at `Permanent`
- Dispatch the generated ops as individual irreversible transactions to establish
  a non-trivial pre-preview state
- Snapshot `state_before = engine.read()` and `hash_before = engine.replay_hash()`
- Build a `preview_tx`: push all ops from the sequence
- Call `engine.dispatch_preview((), preview_tx)`
- `prop_assert_eq!(engine.read(), state_before, "state changed after dispatch_preview")`
- `prop_assert_eq!(engine.replay_hash(), hash_before, "replay_hash changed after dispatch_preview")`
- Verify `lifetimes`/`enabled` isolation INDIRECTLY (they are private fields): after
  dispatch_preview, dispatch the same ops sequence again as individual irreversible txs
  and verify `engine.read()` equals what a fresh engine with same ops (no preview) would show.
  If lifetimes were mutated by preview, subsequent dispatch results would diverge.

**PROP-03** (`prop_03_rule_lifetimes`): two sub-tests, or two separate proptest! blocks
named `prop_03_turns_lifetime` and `prop_03_triggers_lifetime`:

For Turns variant:
- Strategy: `n in 1u32..=10u32`
- Create `CountingRule` with `trigger_count = Rc::new(Cell::new(0))`; clone the Rc to observe
- Add `CountingRule` to engine with `RuleLifetime::Turns(n)`
- Dispatch `n - 1` times using `Transaction::new()` — rule must still be enabled (trigger_count < n)
- Dispatch 1 more time (the nth dispatch) — rule must now be disabled
- Verify: `prop_assert_eq!(trigger_count_clone.get(), n)` — rule fired exactly n times (not n+1 or n-1)
- To confirm disabled after nth: dispatch once more, assert trigger_count_clone.get() == n (no increment)

For Triggers variant:
- Strategy: `n in 1u32..=10u32`
- Create `CountingRule` with shared `trigger_count` Rc
- Add with `RuleLifetime::Triggers(n)`
- Dispatch `n - 1` times — rule must still fire (`before()` called, Triggers decremented each time)
- Dispatch 1 more time (the nth dispatch) — Triggers reaches 0 inside before() loop, rule removed
- Verify: `prop_assert_eq!(trigger_count_clone.get(), n)` — fired exactly n times
- Confirm disabled: dispatch once more, assert trigger_count_clone.get() == n (no further increment)

Note: Turns is decremented in the post-dispatch loop (runs unconditionally). Triggers is
decremented inside the before() loop (only when rule is in enabled set). Both hit enabled.remove
when count reaches 0. The off-by-one test must verify the boundary precisely: n fires, n+1 does not.

**PROP-04** (`prop_04_cancelled_tx_isolation`):
- Strategy: `op_sequence_strategy()`
- Create engine with `NoRule` at `Permanent`
- Snapshot `state_before = engine.read()` and `hash_before = engine.replay_hash()`
- Build `tx = Transaction::new()`, push all ops, then set `tx.cancelled = true`
- Call `engine.dispatch((), tx)`
- `prop_assert_eq!(engine.read(), state_before)` — state unchanged
- `prop_assert_eq!(engine.replay_hash(), hash_before)` — hash unchanged
- Note: Do NOT assert lifetimes are unchanged — `Turns` lifetime decrements unconditionally even
  on cancelled dispatch. PROP-04's invariant is state+hash only. Using `NoRule` with `Permanent`
  avoids this pitfall entirely.
