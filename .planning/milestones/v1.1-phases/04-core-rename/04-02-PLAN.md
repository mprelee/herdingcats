**Internal RuleLifetime:** Since `add_behavior` must still keep behaviors permanently enabled internally (per locked decision: Phase 4 just removes the type from the public API), keep the internal `RuleLifetime` enum as a PRIVATE (non-pub) enum in engine.rs:

```rust
// Internal only — not part of the public API. Kept for Phase 4 compatibility;
// Phase 5 will remove this and replace with per-dispatch is_active() checks.
#[derive(Clone, Copy, Debug)]
enum RuleLifetime {
    Permanent,
    Turns(u32),
    Triggers(u32),

```

This means:
- The engine's `lifetimes: HashMap<&'static str, RuleLifetime>` field continues to exist (private, internal type)
- The engine's `enabled: HashSet<&'static str>` field continues to exist
- The CommitFrame's `lifetime_snapshot` and `enabled_snapshot` fields continue to exist

**CommitFrame:** Update to use `Action<M>` instead of `Transaction<O>`:
```rust
struct CommitFrame<S, M> {
    tx: Action<M>,
    state_hash_before: u64,
    state_hash_after: u64,
    lifetime_snapshot: HashMap<&'static str, RuleLifetime>,
    enabled_snapshot: HashSet<&'static str>,
    _marker: std::marker::PhantomData<S>,
}
```

**Engine struct:** Rename type parameters `O` → `M`, `E` → `I`:
```rust
pub struct Engine<S, M, I, P>
where
    S: Clone,
    M: Mutation<S>,
    P: Copy + Ord,
{
    pub state: S,
    undo_stack: Vec<CommitFrame<S, M>>,
    redo_stack: Vec<CommitFrame<S, M>>,
    behaviors: Vec<Box<dyn Behavior<S, M, I, P>>>,
    enabled: HashSet<&'static str>,
    lifetimes: HashMap<&'static str, RuleLifetime>,
    replay_hash: u64,
}
```
Note: internal field `rules` is renamed to `behaviors`.

**Engine::new:** Update to initialize `behaviors: vec![]`. Same logic otherwise.

**add_behavior method** (replaces add_rule — signature change is breaking/intentional):
```rust
pub fn add_behavior<B>(&mut self, behavior: B)
where
    B: Behavior<S, M, I, P> + 'static,
{
    let id = behavior.id();
    self.behaviors.push(Box::new(behavior));
    self.behaviors.sort_by_key(|b| b.priority());
    self.enabled.insert(id);
    self.lifetimes.insert(id, RuleLifetime::Permanent);
}
```
The `add_rule` method is REMOVED. `add_behavior` always inserts `Permanent` — no lifetime parameter exposed to callers.

**dispatch and dispatch_preview:** Update all occurrences:
- `self.rules` → `self.behaviors`
- `rule.before(...)` → `behavior.before(...)`
- `rule.after(...)` → `behavior.after(...)`
- `rule.id()` → `behavior.id()`
- `tx.ops` → `tx.mutations`
- `tx.irreversible` → The `irreversible` field is GONE from Action. The engine must ALWAYS push a CommitFrame for non-cancelled transactions. Replace the old `if tx.irreversible && !tx.cancelled` guard with simply `if !tx.cancelled`. Every commit is now potentially undoable (Phase 5 will derive irreversibility from mutations).
- The for loop that pushed CommitFrame: no change to its logic except the guard above.
- The `tx.deterministic` check: `Action` still has `deterministic` field — preserve that logic.

Specifically, the hash+push block in `dispatch` changes from:
```rust
if tx.irreversible && !tx.cancelled {
    for op in &tx.ops { ... }
    ...push CommitFrame...
}
```
to:
```rust
if !tx.cancelled {
    for m in &tx.mutations { ... }
    ...push CommitFrame...
}
```
And the hash loop uses `m.hash_bytes()` and checks `tx.deterministic`:
```rust
if tx.deterministic {
    for m in &tx.mutations {
        let h = fnv1a_hash(&m.hash_bytes());
        self.replay_hash ^= h;
        self.replay_hash = self.replay_hash.wrapping_mul(FNV_PRIME);
    }
}
```
(In the current code, hashing is unconditional within the `irreversible` block. Now we need the `deterministic` check to be the gating condition for hashing — which is actually what the semantics should have been. The old code hashed all ops in an `irreversible` tx regardless of `tx.deterministic` — this was a bug that we fix here by separating the deterministic hash path from the commit path.)

**undo:** Update `frame.tx.ops` → `frame.tx.mutations`.

**redo:** Update `frame.tx.ops` → `frame.tx.mutations`.
